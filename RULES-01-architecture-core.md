     5→## Architecture â€” WSS Only, Local-First, Offline-First
     6→
     7→ALL communication between platforms (prtask.com, Android APK, Windows EXE) uses WSS (WebSocket Secure) exclusively. No HTTP GET, POST, PUT, PATCH, DELETE for internal data operations. **No URL paths on prtask.com** â€” WSS handshake at root (`wss://prtask.com`), no `/ws` path. Every CRUD operation, query, mutation, subscription, file upload, diff request, search, sync, and AI inference call goes through WSS messages.
     8→
     9→**Every platform has its own local database (offline-first):**
    10→- **prtask.com** (browser): in-memory lowdb + USearch + automerge (persisted to IndexedDB for offline support)
    11→- **Windows EXE** (Kestrel): lowdb + USearch + automerge â†’ persisted on **local filesystem**
    12→- **Android APK** (MAUI): lowdb + USearch + automerge â†’ persisted on **device filesystem**
    13→
    14→**Real-time sync via WSS + automerge CRDTs:**
    15→All local databases sync with each other through a central WSS sync hub. The Cloudflare Worker Durable Object (production) or a local Kestrel WebSocket server (dev/LAN) acts as the sync hub. When any platform writes data, the change is applied to its local DB first, then synced to the hub via WSS, which broadcasts the automerge CRDT diff to all connected platforms. Offline changes merge automatically on reconnect via automerge conflict-free resolution. Every platform can function fully offline using its local DB.
    16→
    17→**Central sync hub persistence:**
    18→- **Cloudflare mode** (production): Cloudflare Worker Durable Object â†’ lowdb + USearch + automerge â†’ persisted on **R2**
    19→- **Local mode** (dev/LAN): Kestrel WebSocket server â†’ lowdb + USearch + automerge â†’ persisted on **local filesystem**
    20→
    21→IStorageProvider interface abstracts R2 vs filesystem â€” all sync hub persistence goes through IStorageProvider.ReadAsync/WriteAsync/DeleteAsync/ListAsync.
    22→
    23→**Exempted (third-party HTTP only â€” PascalCase callback URLs):**
    24→- Maya payment API (sandbox/production HTTP calls from Worker to Maya servers)
    25→- Didit KYC API (HTTP calls from Worker to Didit servers)
    26→- OAuth provider redirects (Google, GitHub, Azure, Okta â€” browser redirects + Worker token exchange). Callback URLs: `https://prtask.com/GoogleCallback`, `https://prtask.com/GitHubCallback`, `https://prtask.com/AzureCallback`, `https://prtask.com/OktaCallback`
    27→- Maya webhook inbound (Maya POSTs to `https://prtask.com/MayaWebhook`)
    28→- **These PascalCase HTTP paths are the ONLY permitted URL paths on prtask.com.** All other communication is WSS at root.
    29→
    30→**All internal "endpoints" referenced in rules below are WSS message types, not HTTP routes.** Where rules say "GET /api/tasks" or "POST /api/prs", read these as WSS message types: `{type: "query", collection: "tasks"}` or `{type: "mutation", collection: "prs", operation: "create"}`. The WSS gatekeeper handles all WSS messages and executes against lowdb/USearch/automerge. No REST API server exists. RouteConstants defines WSS message type constants, not HTTP paths.
